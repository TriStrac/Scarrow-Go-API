# Scarrow IoT: Connectivity & Pairing Architecture

This document outlines the current state of the Scarrow field deployment ecosystem, the pivot to Bluetooth Low Energy (BLE), and the roadmap for the "Zero-Router" mobile app pairing workflow.

## 1. Where We Are Right Now (Current State)

We successfully pivoted the local communication layer from **Wi-Fi/MQTT** to **Bluetooth Low Energy (BLE)**. 
* **Why the pivot?** This allows the Raspberry Pi to connect to standard Wi-Fi (e.g., a farm network or 4G mobile hotspot) for internet access, while simultaneously maintaining a localized, low-power BLE mesh with the ESP32 nodes.
* **The Pi Hub:** The Go service (`pi-go-service`) is currently deployed and running on the Raspberry Pi via `systemd`. It is actively scanning the area for BLE devices. It buffers incoming data into a local SQLite database and pushes it to `https://scarrow-api.striel.xyz` when internet is available.
* **The ESP32 Nodes:** The firmware has been updated to utilize Deep Sleep. It wakes up when the mmWave radar detects motion, emits the deterrence audio, and briefly broadcasts the event over BLE before going back to sleep.

## 2. Architectural Decision: Who connects to who?

You asked: *"Will the phone send info so the node can connect to the pi? Or will it be reverse?"*

To maximize the ESP32's battery life (since it relies on solar and a 3.7V battery), **The Pi connects to the Node.**

Here is the exact flow for field events:
1. **ESP32 Wakes Up:** The radar triggers. The ESP32 powers on and acts as a **BLE Broadcaster (Peripheral)**. It essentially yells into the void: *"I am Scarrow_Node_01, I saw motion!"*
2. **Pi is Listening:** The Raspberry Pi has abundant power. It acts as the **BLE Scanner (Central)**. It is constantly listening 24/7.
3. **The Handshake:** The moment the Pi hears the ESP32 advertising, the Pi initiates the connection, grabs the telemetry data, writes any pending commands (e.g., "disable_alarm"), and drops the connection.
4. **ESP32 Sleeps:** The ESP32 goes immediately back to deep sleep. 

*If the ESP32 had to act as the scanner and hunt for the Pi, it would drain its battery rapidly.*

## 3. The Future Pairing Workflow (Mobile App Bridge)

To deploy a new system in a remote field without a laptop, we will use a Mobile App as the configuration bridge. 

### Phase A: Hub (Raspberry Pi) Provisioning
When a brand-new Pi is powered on, it doesn't know the local Wi-Fi or its own identity.
1. The Pi boots up and advertises a special BLE service: `Scarrow_Hub_Setup`.
2. The Mobile App connects to the Pi via Bluetooth.
3. The Mobile App makes an HTTP request to the Scarrow Server: *"Register a new Hub and give me an ID."*
4. The Mobile App pushes a JSON payload to the Pi over BLE containing:
   * `wifi_ssid`: The farm/hotspot network name.
   * `wifi_password`: The network password.
   * `hub_id`: The unique ID generated by the server.
5. The Pi saves these credentials, turns off `Scarrow_Hub_Setup` mode, connects to the internet, and begins its permanent job as the BLE Scanner.

### Phase B: Node (ESP32) Provisioning
(Note: One Pi Hub can have many ESP32 Nodes).
1. A brand-new ESP32 is powered on. It advertises `Scarrow_Node_Setup`.
2. The Mobile App connects to the ESP32 via Bluetooth.
3. The Mobile App makes an HTTP request to the Scarrow Server: *"Register a new Node, link it to Hub X, and give me a Node ID."*
4. The Mobile App pushes a JSON payload to the ESP32 over BLE containing:
   * `node_id`: Its unique identity.
   * `hub_filter`: A specific BLE MAC address or Service UUID so the ESP32 knows exactly *which* Pi's connection requests to accept (preventing interference if multiple farms are nearby).
5. The ESP32 saves these to its Non-Volatile Storage (NVS flash), turns off Setup Mode, and enters deep sleep, ready for field duty.

## 4. What To Do Next (Action Items)

To finish the connectivity layer, these are the next development steps:

1. **Update `pi-go-service` for Setup Mode:** Add a localized BLE Peripheral server to the Go code that runs *only* if the SQLite database is missing a `hub_id`. It must expose a writable characteristic to receive Wi-Fi credentials from the phone.
2. **Update `esp32-node` for Setup Mode:** Add logic to `setup()` that checks NVS for a `node_id`. If missing, it stays awake and exposes a writable BLE characteristic to receive its config from the phone.
3. **Develop the Mobile App Flow:** Build the React Native / Flutter screens to handle the BLE scanning, connecting, and JSON payload delivery for both the Hub and the Nodes.
